## 1. Architecture Flow

A user search travels through **two parallel tracks** simultaneously:

```
USER TYPES "Iran" → SUBMIT
          │
          ▼
[SearchIntelligenceController.js]
  POST /api/search { query: "Iran" }
          │
          ▼
[Api::SearchController#create]
  → IntelligenceSearchService.call(query:, user:)
          │
          ├── TRACK A (synchronous, <100ms)
          │   └── Semantic Search (pgvector or ILIKE in demo)
          │       → Returns top 30 cached results
          │       → Response: { cached_results, total_cached, fetching_fresh: true }
          │
          └── TRACK B (async, 5-30s)
              └── FreshIntelligenceJob.perform_later(query:, user_id:)
                      │
                      ▼
                  NewsApiService#fetch_by_query (cached 6h)
                      │
                      ▼
                  GeopoliticalRelevanceFilter
                  (Tier 1: keywords, Tier 2: Gemini LLM)
                      │
                      ▼
                  Article.find_or_create_by(source_url:)
                  → after_create_commit:
                      ├── broadcast_to_globe → ActionCable "globe" channel
                      ├── broadcast_sidebar_update
                      └── FetchArticleContentJob (queued)
                              │
                              ▼
                          AnalyzeArticleJob → AnalysisPipeline
                          → AiAnalysis created
                          → broadcasts color update → "globe" channel
                          → GenerateEmbeddingJob
                          → NarrativeRouteGeneratorService
                              │
                              ▼
                          FreshIntelligenceJob broadcasts completion
                          → ActionCable "intelligence_search_#{query.parameterize}"
                          → { type: "fresh_results_ready", new_articles_count: N }
                              │
                              ▼
                  Frontend receives → re-fetches /api/search + /api/globe_data
                  → Globe re-renders with fresh data
```

## 2. Critical Bugs & Disconnects

### BUG #1 — CRITICAL: ActionCable Channel Name Mismatch (Silent Pipe Break)

**Severity: P0 — Routes and fetch-count broadcasts never reach the globe.**

The `GlobeChannel` subscribes to stream `"globe"`:
```ruby
# app/channels/globe_channel.rb
stream_from "globe"
```

The `Article` model and `AiAnalysis` model correctly broadcast to `"globe"`:
```ruby
ActionCable.server.broadcast("globe", { type: "new_point", ... })
ActionCable.server.broadcast("globe", { type: "update_point", ... })
```

But `FreshIntelligenceJob` and `FetchArticlesJob` broadcast to `"globe_channel"` (wrong name):
```ruby
# app/jobs/fresh_intelligence_job.rb
ActionCable.server.broadcast("globe_channel", { type: "routes_updated", count: N })

# app/jobs/fetch_articles_job.rb
ActionCable.server.broadcast("globe_channel", { type: "articles_fetched", count: N, message: "..." })
```

**Impact:**
- `routes_updated` broadcasts are sent to a channel nobody is subscribed to → silently dropped
- `articles_fetched` broadcasts are sent to a channel nobody is subscribed to → silently dropped
- The globe's `onBroadcast` handler never fires for these two message types
- Route overlays and the recurring fetch notification toast never appear
- The globe only shows new point dots and color updates; multi-hop narrative arcs from FreshIntelligenceJob are invisible

**Fix:** Change both job broadcasts from `"globe_channel"` → `"globe"`.

---

### BUG #2 — HIGH: Globe Never Triggers Route Re-Fetch After Fresh Results

**Severity: P1 — Narrative arcs from a user search never appear on the globe.**

When `fresh_results_ready` arrives on the `IntelligenceSearchChannel`, the frontend:
1. Re-fetches `/api/search` (sidebar refresh — correct)
2. Dispatches `veritas:fresh-results` event (line ~169 in search_intelligence_controller.js)

But there is no code path that triggers a re-fetch of `/api/globe_data` after fresh results arrive. The globe controller listens for `veritas:search` (the initial query event) and `veritas:timelineChange`, `veritas:perspectiveChange`, etc. — but not `veritas:fresh-results`.

**Impact:**
- New narrative arcs generated by `NarrativeRouteGeneratorService` exist in the DB
- The globe never re-fetches `/api/globe_data` to render them as animated arcs
- User sees new dots appearing via ActionCable `new_point` but no arc connections

**Fix:** In `GlobeController`, add a listener for `veritas:fresh-results` that triggers a `_fetchGlobeData()` call.

---

### BUG #3 — HIGH: Loading UX Has No Guaranteed Exit Path on Empty Results

**Severity: P1 — Users can get permanently stuck in the cinematic loading overlay.**

The loading UX timeout is 15 seconds (`search_intelligence_controller.js`, line ~21). After 15 seconds it auto-dismisses. However, there are two failure scenarios with no explicit error path:

**Scenario A — NewsAPI returns 0 results:**
`FreshIntelligenceJob` exits cleanly with `new_articles_count: 0` and broadcasts `fresh_results_ready`. The frontend receives this, calls `_handleFreshResults`, checks `if (newCount > 0)` and skips the re-fetch. This path is correct — the "0 new articles" case is handled.

**Scenario B — `FreshIntelligenceJob` throws an unrescued exception:**
If `NewsApiService` raises (e.g., network timeout, bad API key) before reaching the `broadcast_completion` block, the broadcast at lines 113–120 is never sent. The frontend's `IntelligenceSearchChannel` subscriber never receives `fresh_results_ready`, the loading overlay sits for exactly 15 seconds and then auto-dismisses — but the sidebar still shows "Fetching fresh intelligence…" as a stale notice with no resolution.

**Impact:** Poor UX. On API key exhaustion or network failure, analysts see a 15-second stall with no explanation.

**Fix:** Wrap the entire `FreshIntelligenceJob#perform` body in a `rescue => e` that always broadcasts `fresh_results_ready` with `{ error: e.message, new_articles_count: 0 }`.

### BUG #4 — MEDIUM: `best_journey_data` Serialized to Frontend Under Wrong Key

**Severity: P2 — Globe tooltip/journey panel may display nil or broken journey data.**

`Api::SearchController` serializes articles via `article_json` (lines 41–58). The journey field is:
```ruby
journey_data: article.best_journey_data
```

`best_journey_data` on the Article model returns the result of `narrative_route.as_journey_data` — which is a large nested object containing keys like `routeId`, `totalHops`, `segments`, `propagationSpeed`.

The frontend globe controller processes `cached_results` from the search response and dispatches `veritas:search-results`. Whether the globe's journey panel reads `journey_data` (snake_case) or `journeyData` (camelCase) determines if journeys render. Rails default JSON serialization does **not** camelize keys — so if the globe JS expects `journeyData` (camelCase), it receives `journey_data` (snake_case) and silently gets `undefined`.

**Fix:** Audit `globe_controller.js` for the exact key it reads when handling `veritas:search-results` payload. Either add `.to_camelcase` serialization on the backend, or document that the frontend uses snake_case (and verify consistency).

### BUG #5 — MEDIUM: Race Condition on Concurrent Searches with Same Query

**Severity: P2 — Potential double NewsAPI call and duplicate embedding generation.**

`IntelligenceSearchService#maybe_enqueue_fresh_fetch` reads from a Solid Cache key before deciding to enqueue. But this check is not atomic:

```ruby
cached_call = Rails.cache.read("newsapi:search:#{query.parameterize}:#{Date.current}")
return if cached_call  # already fetched today
FreshIntelligenceJob.perform_later(query:)
# ...cache write happens INSIDE the job, not here
```

If two users search "Iran" within milliseconds of each other before the job runs and writes the cache key, both will enqueue `FreshIntelligenceJob`. Both jobs will fetch from NewsAPI, both will run `find_or_create_by`, and the unique index on `source_url` will prevent DB duplicates — but both will call OpenRouter for relevance filtering and embedding generation on the same articles, wasting API credits.

**Fix:** Write a placeholder cache key immediately in `maybe_enqueue_fresh_fetch` before enqueuing the job (optimistic locking via cache). Use `Rails.cache.write(key, true, expires_in: 6.hours, unless_exist: true)` and only enqueue if the write succeeds.

### BUG #6 — MEDIUM: Embedding Generation Silently Skipped on Content Fetch Failure

**Severity: P2 — Articles may exist in the DB without embeddings, blocking narrative routing.**

The embedding pipeline depends on a sequential chain:
1. `Article` saved → `FetchArticleContentJob` queued
2. Content fetched → `AnalyzeArticleJob` queued
3. AI analysis complete → `GenerateEmbeddingJob` queued
4. Embedding stored → `NarrativeRouteGeneratorService` can find similar articles

If step 2 fails (e.g., source site returns 403/timeout), the chain stops. The article sits in the DB with no `content`, no `ai_analysis`, and no `embedding`. It will never appear in pgvector similarity searches and will never be connected to narrative arcs.

Meanwhile, `FreshIntelligenceJob` calls `EmbeddingService.new.generate(article)` directly (line ~72–82) as a fallback — but `EmbeddingService#generate` requires either `article.content` OR `article.ai_analysis.summary` (line 13), and skips silently if both are nil.

**Impact:** Articles with unfetchable content (paywalls, blocked sites) permanently remain as isolated dots with no narrative routing.

**Fix:** Add a separate embedding path that uses only the headline when both content and summary are unavailable. A headline-only embedding is far better than no embedding.

### BUG #7 — LOW: Heatmap Cluster Cap vs. Point Cap Mismatch Creates Visual Confusion

**Severity: P3 — Analysts see heatmap clusters for countries not visible as points.**

- Globe points: max **200** articles (lines 189–204 of `pages_controller.rb`)
- Heatmap: **all** geolocated articles, weighted
- Heatmap clusters: top **15** countries by article count

An analyst sees a bright red heatmap cluster over a country, but drilling down finds no dots there — because those articles were cut by the 200-point cap but still counted in the heatmap. This silently misrepresents the intelligence landscape.

**Fix:** Either apply the same cap to heatmap data, or document the intentional mismatch in the UI (e.g., "Heatmap shows all data; points show most recent 200").

## 3. UX & Visual Logic Flaws

### FLAW A: Globe Fully Re-Fetches on Every `veritas:search` Event

When a new search runs, `GlobeController` fires `_fetchGlobeData()` which replaces the entire globe dataset. This means:
- All existing arc animations are wiped and rebuilt
- Packet animations reset
- Users lose their current view/zoom state

For a "live intelligence" platform, this is jarring. New articles should be **appended** via ActionCable `new_point` messages (which already works), while the initial globe load only populates the baseline.

**Recommendation:** Make the `veritas:search` event pass a `filter` parameter to `/api/globe_data` rather than rebuilding the entire globe. The globe should _highlight_ matching points, not discard everything else.

---

### FLAW B: No Visual Feedback When Routes Are Generated

After `NarrativeRouteGeneratorService` finishes and the `routes_updated` broadcast fires (which currently goes to the wrong channel — see Bug #1), there is no toast, badge, or animation that tells the analyst "3 new narrative routes were computed." The globe silently gains new arcs on the next re-fetch.

**Recommendation:** Once Bug #1 is fixed, add a brief toast ("3 narrative arcs generated") when `routes_updated` arrives.

---

### FLAW C: `articles_fetched` Message from Recurring Job Has No Frontend Handler

`FetchArticlesJob` broadcasts `{ type: "articles_fetched", count: N, message: "..." }` to `"globe_channel"` (already broken by Bug #1), but even if the channel name were fixed, there is no handler for `"articles_fetched"` in the globe controller's `onBroadcast` switch. The message would be silently ignored.

**Recommendation:** Add a handler for `articles_fetched` in `GlobeController` that shows a live indicator ("15 new geopolitical articles incoming") — this is the real-time heartbeat the platform should display.

---

### FLAW D: IntelligenceSearchChannel Subscriptions Are Never Cleaned Up

After a user searches and `fresh_results_ready` fires, the `IntelligenceSearchChannel` subscription is never unsubscribed. If a user searches 10 times in a session, 10 channel subscriptions accumulate in the browser. Each broadcasts a message, so the handler fires 10 times for each subsequent search.

**Recommendation:** In `SearchIntelligenceController._subscribeToFreshResults`, call `this._freshChannel?.unsubscribe()` before creating a new subscription.

## 5. Pre-GDELT Readiness Assessment

Before adding GDELT (or any new large-scale data source), these pipes must be sealed:

**Must fix before GDELT:**
- Bug #1 (channel name) — GDELT will generate far more `routes_updated` events; silent drops will make the globe completely static
- Bug #2 (globe re-fetch on fresh results) — GDELT articles will never surface as arcs otherwise
- Bug #5 (race condition on concurrent searches) — GDELT at volume will amplify this into costly duplicate API calls

**Should fix before GDELT:**
- Bug #3 (loading UX stuck state) — GDELT pipelines will be slower and timeouts more frequent
- Bug #6 (embedding skip on content failure) — GDELT articles often link to paywalled/blocked sources

**Can coexist with GDELT (address in parallel):**
- Bug #4 (journey_data key casing)
- Bug #7 (heatmap/point cap mismatch)
- Flaws A–D (UX polish)
