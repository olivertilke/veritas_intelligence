# Important Project Guidelines & Decisions

This document contains key technical decisions and architectural rules for the VERITAS project to ensure the AI and the development team remain aligned throughout the build process.

## 1. Frontend Styling
*   **Tailwind CSS:** The project will exclusively use Tailwind CSS for styling the frontend React components. All new UI elements must be built using Tailwind utility classes.

## 2. Backend Routing & Architecture
*   **Nested Resources (Shallow):** We use nested resources for dependent models to show hierarchy, but we keep URLs clean by using `shallow: true`.
    *   `ai_analyses` and `narrative_arcs` are nested under `articles` (since they cannot exist without a parent article).
        *   Collection Example: `GET /articles/:article_id/ai_analyses`
        *   Member Example: `GET /ai_analyses/:id` (not `/articles/:article_id/ai_analyses/:id`)
    *   `countries` are nested under `regions` using the same logic.
*   **Limited Public Actions:** Regular users should never be creating or deleting system-generated data directly (e.g., regions, countries, narrative arcs). These are created by the ingestion mechanics. Therefore, public-facing routes for these resources must be restricted to read-only actions: `only: [:index, :show]`.
*   **Admin Namespace:** All privileged CRUD operations must be wrapped behind an `/admin/...` routing namespace. This namespace is protected by a `before_action` check in an `AdminController` base class that verifies `current_user.role == "admin"`.
*   **Authentication Engine:** We use **Devise** strictly to handle the User models. Sign up, sign in, and sign out routes are automatically generated by `devise_for :users` and should not be manually defined.
*   **Root Path:** The root URL path (`root`) should point to `"dashboard#index"`. This implies we need a `DashboardController` that serves the main intelligence view containing the globe, feed, and panels.

## 3. External API Integrations (GDELT)
*   **Strict Rate Limiting:** The free GDELT 2.0 DOC API enforces a strict rate limit of **1 request every 5 seconds**. 
*   **Error Handling & Retries:** All backend fetchers interacting with GDELT must implement `sleep` intervals (e.g., `sleep 6`) between consecutive queries. They must also rescue from HTTP timeout and Too Many Requests (429) errors using exponential backoff or simply skipping the cycle until the next scheduled worker run. Never allow an API rate limit error to crash the background job queue. #### we are not using GDELT anymore cause it made huge problems, this is why we were implementing NEWSAPI instead, which is now working great!!!!!
